# REFLECTION — Бие даалт 14

## 1) Аль assertion хамгийн их үнэ цэнэтэй байсан бэ? Яагаад?
Энэ ажлын хамгийн үнэ цэнэтэй assertion нь миний хувьд **status code + business rule** хослол байлаа. Зөвхөн `200` ирж байна уу гэдгийг шалгах нь API-ийн чанарыг бүрэн батлахгүй. Жишээ нь `GET /users/{{userId}}` дээр 200 ирсэн ч `email` талбар байхгүй, эсвэл буруу форматтай байвал хэрэглэгчийн талд бодит алдаа гарна. Тиймээс `pm.expect(d.email).to.match(/@/)` зэрэг бизнес дүрмийн баталгаажуулалт нэмэхэд “амьд” contract шалгалт болж өгсөн. Энэ нь integration тестийн зорилготой нийцэж, endpoint бүрийн зөв хариуг зөв утгатай нь хамт шалгах боломж өгсөн.

## 2) Negative test-ийн жишээ, ямар алдааг олох вэ?
Би `GET /users/999999` хүсэлтээр **404** ирэх ёстой кейсийг хийсэн. Энэ negative test нь API буруу id авсан үед зөв алдаа буцааж байгаа эсэхийг батална. Хэрэв сервер санамсаргүйгээр 200 буцаавал client тал “хэрэглэгч олдсон” гэж эндүүрч, цааш буруу UI төлөв эсвэл хоосон өгөгдөл харуулах эрсдэлтэй. Мөн error response-д `message` талбар байгаа эсэхийг шалгаснаар алдаа боловсруулах логик (frontend эсвэл middleware) тогтвортой ажиллах нөхцөл бүрдэнэ. Өөрөөр хэлбэл зөвхөн happy path биш, алдааны замын contract-ыг баталгаажуулж байна.

## 3) Postman дээр pass байсан тест Newman-д fail болсон уу? Яагаад?
Эхний run дээр Postman-д pass байсан зарим тест Newman-д fail болох шалтгаан нь ихэвчлэн **environment тохиргоо** байсан. Ялангуяа `env.ci.json` доторх placeholder credential-үүдийг GitHub Secrets-ээр орлуулахгүй орхиход `Login` request fail болж, түүнээс хамаарах chain request-үүд (token, userId шаарддаг) дарааллаар fail болсон. Энэ нь integration тестэнд өгөгдлийн бэлтгэл (Arrange phase) маш чухал гэдгийг харуулсан. Мөн local болон CI орчинд ижил collection ашиглаж байгаа ч runtime variable өөр байж болно гэдгийг ойлгосон.

## 4) Token/secret-ыг хэрхэн зохицуулсан бэ?
Secret мэдээллийг source control руу commit хийхгүй байх зарчмыг баримталсан. `env.dev.json` болон `env.ci.json` дотор бодит credential-ын оронд placeholder эсвэл сургалтын зорилготой утга үлдээж, CI орчинд GitHub Secrets (`CI_USERNAME`, `CI_PASSWORD`) ашиглан runtime дээр орлуулдаг байдлаар шийдсэн. Token-ийг кодонд хатуу бичээгүй; login response-оос `pm.environment.set("token", ...)` ашиглан динамикаар авсан. Ингэснээр credential эргэлт, аюулгүй байдал, багийн хамтын ажиллагааны үед мөрдөх стандарт хадгалагдсан.

## 5) API өөрчлөгдвөл collection-ийн хамгийн эмзэг хэсэг юу вэ? Яаж бууруулах вэ?
Хамгийн эмзэг хэсэг нь **response schema болон chain variable dependency** гэж үзсэн. Жишээлбэл login response дахь `token` эсвэл `id` талбарын нэр өөрчлөгдвөл дараагийн бүх protected request эвдэрнэ. Мөн `/posts/add` хариуны бүтэц өөрчлөгдвөл `createdPostId` хадгалах логик нурна. Үүнийг бууруулахын тулд:
- Нийтлэг script-ийг collection/folder түвшинд төвлөрүүлэх
- Assertion-уудыг хэт хатуу биш, contract-д төвлөрсөн болгох
- API versioning ашиглах (`/v1`, `/v2`)
- CI дээр pull request бүрт автомат тест ажиллуулах
- Environment, test data, chain variable-уудын нэршлийг нэг стандарттай болгох

Эдгээрийг хийснээр API evolution үед тестийн эвдрэл буурч, засварын зардал багасна.
